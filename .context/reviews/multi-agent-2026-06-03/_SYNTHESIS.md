## Critical

### 1. IOI partial scores are inflated ŌĆö worker early-breaks, server divides by the truncated result count
- File: `judge-worker-rs/src/executor.rs:617-619` (break on first non-AC) feeding `src/lib/judge/verdict.ts:46`
- Dimension: correctness
- Scenario: The Rust worker iterates test cases and unconditionally `break`s on the first non-`Accepted` verdict (`executor.rs:617`), then POSTs only the cases it actually ran. The server is the sole place score is computed: `computeFinalJudgeMetrics` does `score = Math.round((passed * 10000) / results.length) / 100`, where `results.length` is the **truncated** count, not the true total. Example: 10 test cases, pass 1-2, fail 3 ŌåÆ worker sends 3 results ŌåÆ server stores 2/3 = 66.67% instead of the true 2/10 = 20%. The inflated percentage flows through `mapSubmissionPercentageToAssignmentPoints` and `buildIoiLatePenaltyCaseExpr` into the leaderboard, gradebook, live rank, and CSV/JSON exports ŌĆö every IOI assignment/contest/recruiting grade is over-credited, and the inflation grows the earlier the failure occurs. ICPC is unaffected (AC requires `ROUND(score,2)=100`, never produced on a failed run). The unit test `tests/unit/judge/verdict.test.ts` ("2 out of 4 accepted" ŌåÆ 50) masks the bug by hand-building a complete 4-element array including trailing failures that the real worker never emits.
- Suggested fix: Make worker and server agree on the denominator. Preferred: include the true total test-case count in the report payload and have `computeFinalJudgeMetrics` divide by it (the server already has the test cases from the claim and can recompute the denominator). Alternatively remove the early break so the worker reports all cases, or push synthetic non-accepted rows for the remaining cases. Add an integration test that drives the real (truncated) worker output shape through the poll route and asserts the stored score uses the full denominator.

### 2. Cross-instructor IDOR ŌĆö PATCH/DELETE `/problems/[id]` scopes reads to taught groups but lets any `problems.edit`/`problems.delete` holder mutate ANY problem org-wide
- File: `src/app/api/v1/problems/[id]/route.ts:88` (PATCH gate), `:196` (DELETE gate)
- Dimension: authz
- Scenario: The built-in `instructor` role holds `problems.edit`/`problems.delete` but not `groups.view_all`. The read path (`canAccessProblem`, `permissions.ts:140-155`) was hardened so a non-admin can only read another group's private problem if linked to a group they teach ŌĆö but the write path applies **no** group/ownership/visibility scoping; it checks only the global capability (or authorship). The test-case lock at `route.ts:100` only fires when submissions already exist, so an upcoming exam (zero submissions) can have its test data silently altered, destroying exam integrity ŌĆö or the problem DELETEd. This is a read/write authorization asymmetry, not the intended `groups.view_all`-only org-wide-admin model.
- Suggested fix: Add a `canManageProblem(problemId, userId, role)` helper mirroring `canAccessProblem`'s group-scoped branch ŌĆö allow edit/delete only when the actor is the author, holds `groups.view_all`, or the problem is linked (`problemGroupAccess`) to a group the actor teaches (`getAssignedTeachingGroupIds`). - File: `src/lib/audit/events.ts:302-306` (`startAuditEventPruning` no-op); `src/lib/data-retention-maintenance.ts:121-127` (consolidated prune omits `auditEvents`)
- Dimension: data-isolation **(higher signal ŌĆö interlocks with #9 and #10)**
- Scenario: `DATA_RETENTION_DAYS.auditEvents = 90` is defined and `instrumentation.ts:24` calls `startAuditEventPruning()`, but that function is an explicit no-op whose comment falsely claims pruning is handled by `startSensitiveDataPruning()`. The consolidated prune (`pruneSensitiveOperationalData`, verified at lines 121-127) only covers chatMessages, antiCheatEvents, recruitingInvitations, submissions, and loginEvents ŌĆö `auditEvents` is omitted. The only real DELETE against `audit_events` (`cleanupOldEvents` in `db/cleanup.ts`) is reachable solely via `/api/internal/cleanup`, which returns HTTP 410 by default unless `ENABLE_CRON_CLEANUP=true` plus an external cron. In a default deployment nothing ever deletes audit rows. Audit rows hold candidate PII directly (`recruiting-invitations/route.ts:114-116` stores `resourceLabel = candidateName` and `details.candidateEmail`), so candidate names/emails outlive every other retention window indefinitely ŌĆö a silent breach of the documented retention promise (GDPR).

### 4. Permanent account deletion does not erase a recruiting candidate's PII on the `recruiting_invitations` row (GDPR right-to-erasure gap)
- File: `src/app/api/v1/users/[id]/route.ts:473` (`db.delete(users)`); schema `src/lib/db/schema.pg.ts:966-974` (`candidateName`/`candidateEmail`/`ipAddress`/`metadata`), `:970-971` (`userId onDelete: set null`)
- Dimension: data-isolation **(higher signal ŌĆö interlocks with #3 and #10)**
- Scenario: On `DELETE ?permanent=true` the handler runs only `db.delete(users)`. The invitation FK on `userId` is `onDelete: 'set null'`, so the invitation row survives with `candidateName` (NOT NULL), `candidateEmail`, `ipAddress`, and `metadata` (which can hold recruiter-supplied custom PII via `z.record(string,string)`) intact ŌĆö only the user link is nulled. Candidate name/email were copied onto the user row at redeem (`recruiting-invitations.ts:662-671`), so erasing the user does not erase the source-of-record copy. Worse, there is no admin path to delete a redeemed invitation: the invitation DELETE route rejects any non-pending status (`recruiting-invitations/[invitationId]/route.ts:186-188`). The only removal is the `updatedAt`-keyed `pruneRecruitingInvitations` sweep, which any row touch (metadata edit, password reset) postpones indefinitely. A GDPR erasure fulfilled via "permanently delete user" leaves the candidate's PII in the DB. (Note: the lib path is `src/lib/assignments/recruiting-invitations.ts`, not the `src/lib/recruiting/` cited in the original finding.)
- Suggested fix: In the permanent-delete branch, within the same transaction, also scrub the candidate's invitation PII ŌĆö delete the `recruiting_invitations` rows for that `userId`, or null/replace `candidateName`/`candidateEmail`/`ipAddress` and strip non-`_sys` metadata while keeping an audit shell. Relax the invitation DELETE route (or add an erase endpoint) so a redeemed candidate's PII can be removed on request, and record the erasure as an audit event.

### 5. Stale-claim reclaim leaks the original worker's `active_tasks` counter (permanent capacity loss)
- File: `src/lib/judge/claim-query.ts:38-87` (`worker_bump` CTE at `:79-85`; reclaim condition `:45-47`)
- Dimension: reliability
- Scenario: `buildClaimSql` reclaims any `queued`/`judging` submission whose `judge_claimed_at` is older than `staleClaimTimeoutMs`, regardless of whether the previous owner is alive. The `worker_bump` CTE increments `active_tasks` only for the NEW claiming worker; it never decrements the previous owner. The only decrement paths are the token-guarded final-verdict poll and the staleness sweep (which fires only when a worker stops heartbeating). Failure mode: a worker hangs on one submission (runaway/stuck container it cannot kill) but its heartbeat thread keeps reporting `online`. Another worker reclaims after timeout; the original worker's `active_tasks` stays incremented forever ŌĆö its late poll is token-fenced and rolls back (never decrements), and the sweep never reaps a still-heartbeating worker. Each event permanently reduces effective capacity (`active_tasks < concurrency` gate); after enough events the healthy worker silently stops accepting work, throttling the whole fleet during a contest/exam. Heartbeat self-reported `activeTasks` is intentionally not persisted, so it cannot self-correct. The only remedy is admin force-DELETE.
- Suggested fix: When reclaiming a stale submission whose `judge_worker_id` is set and differs from the claiming worker, atomically decrement the previous owner in the same statement ŌĆö capture `candidate.previous_worker_id` and add `UPDATE judge_workers SET active_tasks = GREATEST(active_tasks - 1, 0) WHERE id = (SELECT previous_worker_id FROM candidate) AND previous_worker_id IS DISTINCT FROM @workerId`. Alternatively, periodically reconcile each worker's `active_tasks` against its actual owned `queued`/`judging` count.

### 6. Discussion thread lists over-fetch ALL post rows just to compute reply counts
- File: `src/lib/discussions/data.ts:88-204` (e.g. `:93`, `:136`, `:181`, `:196`)
- Dimension: perf
- Scenario: `listGeneralDiscussionThreads` (limit 50), `listAllProblemDiscussionThreads` (limit 200), `listProblemDiscussionThreads`/`listProblemSolutionThreads` (limit 50) all eager-load the full `posts` relation with `posts: { columns: { id: true } }`. Drizzle materializes every post row for every thread in the page; the only downstream use is `thread.posts.length` as a reply-count label. The community "Problem talk" tab (`scope=problems`) is a public, guest-reachable route that loads up to 200 threads ├Ś all their posts ŌĆö and visibility filtering runs in JS *after* the fetch, so the DB fetch is unbounded regardless of viewer scope. There is no per-thread post cap, so a few popular threads with thousands of comments turn one page load into hundreds of thousands of rows pulled into Node, spiking memory and latency, while the rendered output is just a number.
- Suggested fix: Replace the eager `posts` relation with a batched aggregate count ŌĆö one `SELECT thread_id, COUNT(*) FROM discussion_posts WHERE thread_id IN (...) GROUP BY thread_id` (mirroring `listVoteSummaries`) mapped onto threads. Keep eager post loading only where post bodies are rendered (`getDiscussionThreadById`, `listProblemEditorials`). The `dp_thread_idx` index already supports the grouped count.

### 9. Assignment PATCH does not invalidate the leaderboard ranking cache ŌĆö `latePenalty`/deadline/points/scoringModel edits leave stale standings that diverge from the gradebook
- File: `src/app/api/v1/groups/[id]/assignments/[assignmentId]/route.ts:197-214` (PATCH, no `invalidateRankingCache`); `src/lib/assignments/management.ts:225-317`
- Dimension: correctness (compounds #1, #10 on the IOI scoring surface)
- Scenario: `computeContestRanking` caches per-assignment rankings in an LRU (30s TTL). ### 11. `computeContestAnalytics` fetches all submission rows for an assignment with no LIMIT
- File: `src/lib/assignments/contest-analytics.ts:182-188` (`allAcSubs`) and `:242-253` (`studentProgressions` `submissionRows`)
- Dimension: perf
- Scenario: Analytics issues `SELECT ... FROM submissions WHERE assignment_id=@id AND ROUND(score,2)=100 ORDER BY submitted_at` (all full-score rows) and, when `includeTimeline=true` (the route always passes true), `SELECT ... FROM submissions s JOIN users/assignment_problems/exam_sessions WHERE s.assignment_id=@id ORDER BY submitted_at` ŌĆö every submission row for the contest, unbounded, joined and iterated in app memory to build per-user progressions. For a large/long-running contest (thousands of participants ├Ś dozens of resubmissions) this is a multi-hundred-thousand-row fetch plus O(rows) JS work. The 60s LRU + 30s stale-while-revalidate cache amortizes frequency, but a cache miss / first instructor view / eviction (max 100 keys) pays the full unbounded cost synchronously, and the cached object retains all progression arrays in memory. Access is instructor-only and rate-limited, so this is an authorized-instructor perf hazard, not anonymous DoS.
- Suggested fix: Push the progression and first-AC aggregation into SQL (per-user running best per problem via window functions / `DISTINCT ON`) so only O(users ├Ś problems) rows return, or cap per-user progression points. At minimum add an explicit LIMIT guard and a `truncated` flag so a pathological contest cannot OOM the app server on a cache miss.

### 14. Judge worker `/register` has no rate limit or dedup ŌĆö a shared-token holder can flood `judge_workers`
- File: `src/app/api/v1/judge/register/route.ts:24-76`
- Dimension: security
- Scenario: The register endpoint is gated only by the optional IP allowlist (allow-all when `JUDGE_ALLOWED_IPS` is unset ŌĆö the default) plus the shared `JUDGE_AUTH_TOKEN`. Unlike `claim/route.ts` (which calls `consumeUserApiRateLimit`), register calls no rate limiter and each successful POST unconditionally INSERTs a brand-new `judge_workers` row with no per-hostname dedup or cap (no unique constraint/index on hostname). Anyone holding the shared token (leaked CI/compose env, backup, compromised worker host) can issue unbounded register calls, bloating the table ŌĆö each row counted as `online` by the staleness sweep and admin inventory, and never deleted (the sweep only transitions onlineŌåÆstaleŌåÆoffline). This is a DoS/resource-exhaustion and audit-noise vector, not verdict forgery.
- Suggested fix: Add a `rateLimit` bucket keyed on client IP (and/or hostname) to the register handler, and/or upsert by `(hostname, ipAddress)` instead of always inserting. Optionally cap the number of non-deregistered `online` workers per hostname and reject excess registrations.
