### Refined finding ‚Äî DOC-25 (`AUTH_TRUST_HOST` default)

The underlying code changed since the last review (commit `b2397f5b`, "fix(auth): default AUTH_TRUST_HOST=false and canonicalize X-Forwarded-Host"):

- `docker-compose.production.yml:115` fallback is now `AUTH_TRUST_HOST=${AUTH_TRUST_HOST:-false}` (was `true`).
- `docs/deployment.md:44` table already says "Default: `false`" ‚Äî this row is now *technically* accurate for the compose fallback.
- **But** `deploy-docker.sh:761` (fresh-generation), `:918` (`ensure_env_literal AUTH_TRUST_HOST true`, applied even to an *existing* `.env.production`), and `:1000` (`upsert_env_literal AUTH_TRUST_HOST true`) all unconditionally force `AUTH_TRUST_HOST=true` on every deploy that goes through the standard script ‚Äî with an explicit code comment at `:912` ("AUTH_TRUST_HOST must be true in production: current Auth.js rejects requests‚Ä¶"). `docs/deployment.md:18`'s example `.env.production` also shows `AUTH_TRUST_HOST=true`.
- **Net:** the compose-fallback default is `false`, but the *effective* production default (via the only supported deploy path) is always `true`. The table's "Default: `false`" is misleading in context ‚Äî a reader who takes it at face value doesn't learn that `deploy-docker.sh` overrides it every time. Recommend adding a note to the `AUTH_TRUST_HOST` row: "compose fallback is `false`, but `deploy-docker.sh` always forces `true` in production ‚Äî see line 18 example." Downgraded from the prior MEDIUM "contradiction" framing to a **LOW/MEDIUM "needs clarifying footnote"** since the raw contradiction between two docs is gone, but the doc-vs-effective-behavior gap remains.
