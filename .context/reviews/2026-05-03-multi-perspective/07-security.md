# Security Review — Adversarial Perspective (Post-Remediation)

**Reviewer**: Security Researcher / Attacker
**Date**: 2026-05-03
**Scope**: Full-stack security audit, focused on new findings and verification of previously reported items
**Context**: Platform used for recruiting coding tests, student exams, and programming contests

---

## Executive Summary

JudgeKit has undergone **significant security hardening** since the April 17 review. Of the 14 vulnerabilities reported (1 CRITICAL, 4 HIGH, 6 MEDIUM, 3 LOW), **11 are now fixed or mitigated**. The remaining 3 are HIGH items (MFA, lockdown browser, candidate name in resume path) and 1 MEDIUM (anonymized export).

The platform's security posture is now **above average for a self-hosted OJ**. The auth/crypto/sandbox/CSRF stack is solid. The anti-cheat architecture is now defensible for honor-system use cases. The remaining HIGH items are feature gaps (MFA, lockdown browser), not implementation bugs.

## New findings

### N1. [MEDIUM] Anti-cheat heartbeat can be spoofed with browser automation

**Where**: `src/app/api/v1/contests/[assignmentId]/anti-cheat/route.ts`
**Attack**: A candidate using a headless browser (Puppeteer/Playwright) can automate:
1. Open the exam page in headless Chrome
2. Script the anti-cheat monitor to send heartbeats and focus events
3. Use the same session to submit code generated by ChatGPT on another device

The heartbeat enforcement checks that an event exists within 60 seconds — it doesn't verify that the event came from a genuine browser interaction vs. scripted automation. A `document.dispatchEvent(new Event('focus'))` sent on a timer produces an identical anti-cheat event to a real tab switch.

**Impact**: The curl-only attack is closed, but a scripted browser attack produces an identical "clean session" that reviewers cannot distinguish from honest behavior.

**Fix**: Add a cryptographic challenge-response to heartbeat events. The server issues a per-session nonce; the browser must compute a response (e.g., HMAC with the nonce) that headless browsers can also compute but which ties the event to the specific browser session. This doesn't fully prevent automation but raises the bar above "send a POST on a timer."

### N2. [MEDIUM] Docker socket proxy `CONTAINERS=1` still allows container inspection

**Where**: `docker-compose.production.yml` socket proxy config
**Attack**: Even with `CONTAINERS=1` only, the socket proxy allows:
- `GET /containers/json` — list all running containers, their names, environment variables, and mounts
- `GET /containers/{id}/json` — inspect a specific container's full configuration

A compromised judge worker container can enumerate all other containers on the host, discover environment variables (including `JUDGE_AUTH_TOKEN` if it leaks into a container env), and identify the database container.

**Fix**: Restrict the socket proxy to `CONTAINERS=1` for create/start/stop only, not for GET/inspect. Or use `ALLOW_RESTARTS=0` and `POST=0` to further restrict.

### N3. [LOW] Recruiting token entropy

**Where**: `recruiting-invitations.ts` token generation
**Analysis**: The recruiting invitation token is a `nanoid(32)` (base64url), giving ~191 bits of entropy. This is more than sufficient against brute-force — even at 10,000 requests/second, enumeration is infeasible within the token's validity period.

However, the token is stored as a SHA-256 hash in `recruiting_invitations.tokenHash`. The original token is never stored, which is good. But the token appears in URLs (browser history, server logs, link previews), and the `candidateName` + `candidateEmail` are stored in plaintext alongside the hash. If the `recruiting_invitations` table is compromised, the attacker gets candidate PII even without the token.

**Fix**: Consider encrypting `candidateName` and `candidateEmail` at rest (application-level encryption with a key not in the database).

### N4. [LOW] Submission cancel window creates a timing attack surface

**Where**: `5f7fbef5` — 4-second cancel window for submissions
**Analysis**: The 4-second cancel window allows a student to submit, see the judge output, and cancel if the result is unfavorable. While the cancel is audited, the student gains an information advantage — they see partial results (compilation success/failure, first test case outcome) before committing to the submission.

For exam-mode assignments, this could be used to "probe" the judge with throwaway submissions. The rate limiter (max submissions per minute) limits this but doesn't eliminate it.

**Fix**: For exam-mode assignments with `enableAntiCheat`, consider disabling the cancel window or making it shorter (1-2 seconds).

## Attack scenario re-evaluation

### Scenario 1: Candidate Cheats with ChatGPT (second device)
**April assessment**: Candidate scores 100% with zero detection.
**May assessment**: Candidate can still use ChatGPT on a second device while the browser monitor shows perfect heartbeats. However:
- The heartbeat is now enforced server-side — they must keep the browser open and monitored
- The honest AI disclosure tells candidates that AI use is "difficult to detect" — managing expectations
- Code similarity catches candidates who copy from each other but not from ChatGPT
**Verdict**: Still possible, but the platform is now honest about the limitation. For low-stakes screening, this is acceptable. For high-stakes, add SEB/proctoring.

### Scenario 2: Leaked Backup Exposes Test Data
**April assessment**: Full system compromise.
**May assessment**: Backups now nullify password hashes, session tokens, and API keys even in full-fidelity mode. A leaked backup no longer provides immediate credential reuse. Recruiting invitation token hashes and judge worker claim tokens are still in sanitized-only exports (not ALWAYS_REDACT).
**Verdict**: Significantly mitigated. A leaked backup is still sensitive (contains submission source code, problem data, candidate PII) but no longer provides direct credential access.

### Scenario 3: Compromised Worker Container
**April assessment**: Build malicious image, access all test data.
**May assessment**: The shared JUDGE_AUTH_TOKEN fallback is removed, compile swap is capped, dockerfilePath is anchored, and seccomp is applied to compile. The socket proxy still allows container inspection (N2), but building malicious images is no longer possible with `CONTAINERS=1` only.
**Verdict**: Significantly harder. A kernel exploit from the compile sandbox is still theoretically possible but requires a 0-day in the host kernel.

## Summary

JudgeKit's security posture improved from "good foundations with critical operational gaps" to "solid security engineering with documented architectural limits." The 11 fixes since April are genuine — not band-aids. The heartbeat enforcement, backup redaction, and removed shared judge token are the three highest-impact changes.

The remaining HIGH items (MFA, lockdown browser) are feature gaps that require dedicated engineering sprints, not bug fixes. The new findings (N1-N4) are MEDIUM/LOW and represent incremental hardening, not architectural ceilings.

For recruiting and exam use cases, the platform is now **defensible for honor-system scenarios** and **honestly documents its limitations** for high-stakes scenarios. This is the right posture for a self-hosted platform — acknowledge the ceiling, mitigate what you can, and let operators add external controls (SEB, proctoring) for what you can't.
